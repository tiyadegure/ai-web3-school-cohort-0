## 核心定义

Verifiable AI 关注的是：当 AI 输出会影响资产、权限、声誉或治理时，我们能否验证它的输入、模型、执行环境、推理过程或至少留下可审计证据。

**一句话：** 让"相信模型"变成"验证证据和约束"。

---

## 第一性原理

验证成本必须和输出影响力匹配。

- 先验证来源：输入、模型版本、服务方、执行时间
- 再验证过程：是否在可信环境中执行，是否可重放或可证明
- 最后验证结果影响：输出能改变什么链上状态，是否需要挑战期

---

## 知识节点

### ⭐⭐ TEE

用可信执行环境隔离代码和数据，通过 attestation 证明某段程序在特定环境中运行。

**适用场景：** 私有数据评分、模型推理、Agent runtime 证明

**强项：**
- 工程上相对可用：能跑复杂程序、处理私有输入、证明成本低于 ZK
- 适合"不能公开输入或模型细节，但需要证明程序在受保护环境运行过"

**弱点：**
- 不是纯密码学信任——硬件漏洞、远程证明服务、供应链都是信任假设
- 文档要写清楚信任谁、证明什么、不证明什么

### ⭐⭐⭐ ZK

允许证明某个计算满足条件，而不泄露全部输入或重新执行完整计算。

**优势：** 密码学验证强
**劣势：** 生成证明成本高、工程复杂

**适用边界：**
- 适合证明边界清楚、计算规模可控的任务（小模型分类、后处理规则、数据聚合）
- 对 LLM 来说，完整证明每一步 token 推理通常还不现实
- 很多产品选择证明关键部分，而不是证明整个模型

### ⭐⭐⭐ zkML

把机器学习推理转成可证明计算。

**适用场景：** 模型较小、结构固定、输出需要强验证（小型风控模型、资格判断、分类阈值判断）

**设计前先问：**
- 是否真的需要隐藏输入？
- 是否真的需要链上验证？
- 能否接受证明延迟？
- 模型能否转成电路？

很多"需要可信 AI"的场景，其实用签名日志、TEE 或人工复核更经济。

### ⭐⭐⭐ Proof of Inference

证明某个输出确实来自某个模型、输入和执行环境。

**实现路线选择：**

| 证明需求 | 推荐方案 |
|----------|----------|
| 证明服务返回过这个结果 | 签名日志 |
| 证明结果来自特定二进制和模型 | TEE |
| 最小化信任第三方 | ZK（更贵） |

**关键原则：** 产品文档应该明确证明对象——证明输入没变、证明模型版本、证明运行环境、证明输出未篡改、证明完整计算正确，这几个目标不是同一件事。

### ⭐⭐ Verifiable Compute

让链下计算结果能被链上或第三方验证。

**模式：** 链下执行 + 链上验证（摘要/证明/签名结果）

**关键原则：**
- 既保留链上可验证性，又避免把复杂推理直接塞进合约
- 注意数据可用性：链上只有 hash/proof 时，用户需要知道原始输入输出在哪里

### ⭐⭐ Benchmark

用来比较模型或 Agent 的能力，但不能替代任务级验证。

**关键原则：**
- 公开 benchmark 说明一般能力，不能证明某次输出正确
- 通用榜单很好但在你的链上任务里可能经常出错（读错 decimals、忽略 chain id）
- 项目应该建立自己的 benchmark
- AI × Web3 benchmark 应包含：正常样本 + 边界样本 + 攻击样本（错误链、恶意上下文、过期价格、同名 token、revert 交易、假合约文档）

### ⭐⭐ Audit Trail

最基础、最实用的可验证层。

**记录：** 输入、输出、模型版本、工具调用、时间、用户确认、交易哈希、错误

**关键原则：**
- 最容易落地的可验证 AI 起点
- 不能证明模型绝对正确，但能证明系统当时看到了什么、调用了什么、用户确认了什么
- 敏感原文加密保存，公开层只放 hash、摘要和权限可控的引用

## 成熟系统的混合验证策略

```
日志（日常复盘）→ Attestation（服务证明）→ ZK/TEE（高风险场景）→ Challenge（处理争议）
   ↑ 最容易落地                                    最强验证 ↑
```

**核心洞察：** 不是每个场景都需要 ZK——按风险分层选择验证强度。Audit Trail 是起点，TEE 是实用方案，ZK 是终极目标但成本高。

---

## 与 Bridge 知识链的关系

```
Chain-aware Context → Web3 Tool Use → Agent Workflow
→ Agent Wallet → Machine Payment → Settlement & Escrow
→ Agent Identity → Agent Trust & Reputation → AI Oracle → Verifiable AI
                                                                  ↓
                                                          所有层的验证基础
```

**核心洞察：** Verifiable AI 是 AI Oracle、Agent Trust、Settlement 和 Governance AI 的基础。它不一定总是链上证明，但必须让关键输出有可追溯证据。

| Agent Trust & Reputation | Attestation 和 Validation 是信任的验证层 |

| 主方向 Privacy | TEE 和 ZK 是隐私保护的核心技术 |
| Agent Wallet | 签名日志是 Agent 操作的审计轨迹 |

---

## 一句话总结

验证成本必须匹配输出影响力：低风险留日志、中风险用 TEE、高风险用 ZK。Audit Trail 是最容易落地的起点，不是每个场景都需要最强证明。
