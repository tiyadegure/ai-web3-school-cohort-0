## 核心定义

AI Oracle 是把模型输出、评分、分类或推理结果提交给链上系统使用的机制。它的问题不只是"如何上链"，而是如何记录输入、模型、版本、证明和争议。

**一句话：** AI Oracle 的核心，不是让模型替合约做判断，而是让模型判断变成可记录、可验证、可挑战的输入。

---

## 第一性原理

当 AI 输出会影响链上资产或权限时，输出本身必须有来源、边界和争议机制。

- 输入要可追溯：模型看到了什么，不能只留最终答案
- 结果要结构化：链上不适合消费长篇自然语言，需要明确字段
- 争议要提前设计：谁能挑战，挑战期多久，如何复核

## 知识节点

### ⭐⭐ AI Output

模型给出的结果（文本/标签/分数/结构化 JSON/决策建议）。

**关键原则：**
- 链上系统最好只消费结构化输出（accepted: true, riskScore: 72），长文本存链下用 hash 关联
- 分成"机器字段"和"人类解释"两层——机器字段进合约，人类解释进 UI/报告/争议材料
- 如果输出影响资金或权限，记录 confidence、模型版本、输入 hash、输出 hash、生成时间

### ⭐⭐ Data Feed

持续提供模型处理后的数据（地址风险评分、内容审核标签、交易意图分类等）。

**关键原则：**
- AI data feed 比价格 feed 更容易漂移——模型升级、训练数据变化、prompt 调整都会导致评分变化
- Feed 应该明确版本，并允许查询历史
- 用于自动执行时，至少检查 stale data、异常跳变和来源签名

### ⭐⭐ Model Result

需要包含模型版本、prompt/任务模板、输入引用和输出字段。

**关键原则：**
- 只保存结果不保存生成条件，后续很难复核
- 输出 schema 要明确（风险评分 0-100，越高代表风险越高还是越低）
- 多模型系统还要记录路由信息（为什么选这个模型、是否用了 fallback）

### ⭐⭐⭐ Oracle Risk

AI Oracle 风险包括：模型错误、输入污染、prompt injection、数据偏差、执行环境被篡改、输出不可复核、经济攻击。

**关键原则：**
- 风险取决于输出影响什么——展示标签（低风险）vs 释放资金/罚没 stake（高风险）
- 按影响范围分层：低风险直接展示、中风险人工复核、高风险需要 challenge + multi-evaluator

### ⭐⭐ Attestation

证明某个主体或执行环境对 AI 输出作出了声明（TEE 证明运行环境、验证者证明通过测试等）。

**关键原则：**
- Attestation ≠ 结果正确，但能证明"谁在什么条件下给出了这个结果"
- 字段要具体：验证者是谁、验证对象是什么、输入输出 hash、模型版本、过期时间、是否可撤销
- 作为可消费信号，合约可要求有某类 attestation 才进入下一步，但仍保留争议窗口

### ⭐⭐⭐ Proof of Inference

证明某个输出确实来自某个模型和输入。

**实现路线：** TEE、ZK、签名日志、可重放推理、可信服务证明

**关键原则：**
- 先明确"证明什么"：用了某个模型？输入没变？输出由某环境生成？推理过程完整正确？
- 大型 LLM 的完整推理证明目前成本很高
- 现实系统采用折中：TEE 证明运行环境 + 签名日志证明输入输出 + ZK 证明关键后处理 + challenge 处理争议

### ⭐⭐⭐ Dispute / Challenge

对 AI Oracle 输出提出异议的机制。

**关键原则：**
- 采用 optimistic 模式：先接受结果，给出挑战窗口，有人提出证据则进入复核
- 没有 challenge 机制的 AI Oracle，很容易把模型错误硬写进链上状态
- 按金额和风险分层：小额短窗口、高价值长窗口、高争议进人工/多方 evaluator

## 与 Bridge 知识链的关系

```
Chain-aware Context → Web3 Tool Use → Agent Workflow
→ Agent Wallet → Machine Payment → Settlement & Escrow
→ Agent Identity → Agent Trust & Reputation → AI Oracle
                                                  ↓
                                          Oracle + Verifiable AI + Settlement 交叉点
```

**核心洞察：** AI Oracle 位于 Oracle、Verifiable AI、Settlement & Escrow 的交叉点。最稳妥的路径是从低风险场景开始（标注、摘要、预警），涉及资金释放/惩罚/权限变更时，需要证明、挑战期和人工复核。

## 与已有知识的关联

| 已学章节 | 与 AI Oracle 的关系 |
|----------|---------------------|
| Settlement & Escrow | AI Oracle 可以作为 Evaluator——"任务是否完成"的链上判断 |
| Agent Trust & Reputation | Attestation 和 Validation 为 Oracle 输出提供验证信号 |
| Machine Payment | Oracle 输出可能触发支付/退款——错误输出直接造成资金损失 |
| Agent Identity | Oracle 需要身份——谁提供的输出，谁来验证 |
| 主方向 Privacy | Proof of Inference 的 TEE/ZK 路线与隐私保护直接相关 |

---

## 一句话总结

AI Oracle 不是"让模型上链"，而是让模型判断变成可记录、可验证、可挑战的输入。核心设计：结构化输出 + 版本追踪 + Attestation + Challenge Window + 按风险分层。
