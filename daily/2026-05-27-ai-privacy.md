## 核心定义

AI Privacy 关注的是：用户数据、钱包身份、链上行为、私有记忆、API key 和模型上下文之间，哪些可以共享，哪些必须隔离，哪些只应该在本地处理。

**一句话：** AI × Web3 的隐私问题，不是单点数据泄漏，而是链上公开身份和 AI 私有上下文被拼接。

---

## 第一性原理

模型只应该看到完成任务所需的最小数据。

- 数据边界要显式：哪些进模型，哪些只进工具，哪些只在本地
- 记忆要可管理：用户要能查看、删除、导出或关闭记忆
- 链上身份要谨慎关联：不要默认把多个地址和真实身份合并展示

## 知识节点

### ⭐⭐ Data Boundary

定义数据在用户设备、应用后端、模型服务、链上和第三方工具之间如何流动。

**关键原则：**
- 每个数据字段都要问：是否必要、是否可脱敏、是否可本地处理、是否会进日志、是否会给第三方模型
- 最好画成数据流图，而不是写成隐私政策长文
- Agent 的边界按工具划分：浏览器工具、钱包工具、模型工具、索引工具看到的数据不应该默认互通

### ⭐⭐ Local AI

把部分推理放在用户设备或私有环境中，减少敏感数据外发。

**关键原则：**
- 好用原则：先在本地做筛选和脱敏，再把必要摘要发给强模型
- 例如本地先从钱包历史中提取"与本任务有关的 3 笔交易"，而不是上传全部历史
- 本地不等于绝对安全——本地 Agent 仍可能被恶意文件/插件/网页攻击

### ⭐⭐ Private Memory

Agent 为用户保存的长期偏好、历史任务和私有上下文。

**关键原则：**
- 不能变成不可见黑箱——用户应该知道记了什么、为什么记、如何删除
- 分层：短期任务记忆、长期偏好、敏感身份信息、钱包相关记录
- 常见风险："为了更智能而永久保存一切"——默认应该最小化、可查看、可删除

### ⭐⭐⭐ Secret Management

处理私钥、API key、session token、JWT、wallet credentials 和加密密钥。

**底线：** 不进 prompt、不进模型输出、不进普通日志、不进 analytics

**关键原则：**
- 通过受控工具使用 secret，不是让模型直接读取
- 要处理轮换和撤销：API key 泄漏后能否快速换掉？session token 是否短期有效？
- Web3 特别要求：助记词和私钥不应该出现在任何 Agent prompt，即使用户主动粘贴，系统也应该识别并拒绝处理

### ⭐⭐ Minimal Disclosure

只暴露完成任务所需的最少信息。

**例子：**
- 证明"余额足够支付 10 USDC"，不暴露全部持仓
- 证明"用户有权限"，不公开真实身份
- 总结交易风险，不上传完整钱包历史

**实现方式：** 只传必要字段、零知识证明、一次性地址、按应用隔离身份、summary 替代原文

### ⭐⭐ Encrypted Data

保护存储和传输中的隐私，但不能自动解决模型使用时的泄漏。

**关键原则：**
- 数据进入模型推理时通常需要被解密或在可信环境中处理
- 加密要和访问控制、密钥管理、TEE、最小化上下文结合
- 谁持有密钥？平台持钥/用户持钥/多方分片/硬件持钥 → 信任模型完全不同
- "服务器加密保存但每次推理都解密发给模型" → 解决的是存储安全，不是模型使用隐私

### ⭐⭐ User Consent

让用户明确知道数据会被怎样使用。

**关键原则：**
- 同意连接钱包 ≠ 同意分析全部历史
- 同意使用 Agent ≠ 同意把数据发给所有第三方模型
- 同意保存记忆 ≠ 同意永久保存
- 应该具体到动作和数据类型，每个是不同开关
- 应该可撤销——撤销后说明哪些数据会删除，哪些链上记录无法删除

## 隐私的核心风险

```
链上公开数据（地址/交易/持仓）
         +
AI 私有上下文（聊天/偏好/记忆/工具调用）
         ↓
    真实身份画像 ← 这才是隐私风险的核心
```

**核心洞察：** 单看某个地址可能只是公开信息，但和聊天记录、邮箱、地理位置、社群身份结合后，就可能变成真实身份画像。

## 一句话总结

AI × Web3 隐私 = Data Boundary（数据边界）+ Local AI（本地推理）+ Private Memory（可管理记忆）+ Secret Management（密钥安全）+ Minimal Disclosure（最小披露）+ Encrypted Data（加密）+ User Consent（用户同意）。核心风险是链上公开身份 + AI 私有上下文被拼接成真实身份画像。
