## 核心定义

Machine Payment 讨论的是 Agent、API、服务和钱包之间如何自动完成报价、授权、付款、收据和预算控制。

**一句话：** 机器支付的核心，是把"付款意图"和"实际结算"拆开，并让每一步都有凭证。

**第一性原理：** Agent 不应该拥有无限支付能力，只应该拿到具体任务、预算和收款方范围内的支付权限。

## 8 个知识节点

### ⭐ Stablecoin Payment（稳定币支付）
稳定币适合机器支付：价格稳定、结算快、可编程、易验证。

Agent 必须区分：
- **计价币种 vs 结算币种**（USD 报价，USDC 支付）
- **token address、decimals、chain id**
- **approve 本身是高风险动作**，不能和普通付款混在一起

判断稳定币支付方案至少看四件事：
1. 付款 token 在目标链上有足够流动性
2. 收款方接受该 token
3. 支付失败是否会消耗 gas
4. 用户是否需要提前 approve

### ⭐ Budget（预算）
用户授权给 Agent 的最大支出范围。

**预算应拆成多层：**
- 全局预算
- 任务预算
- 单次调用上限
- 服务方上限
- 紧急停止条件

**常见错误：** 只设总预算，不设频率和服务方范围。结果 Agent 可以把全部预算花在一个异常报价上。

**预算系统应能回答：** 这笔付款是否属于当前任务？是否在授权服务范围内？是否超过频率或金额限制？

### ⭐⭐ Quote（报价）
服务方给 Agent 的可执行报价。

**合格 quote 至少包含：** 服务内容、价格、币种、收款地址、有效期、交付条件、退款条件、quote id。

**有效期很重要：** 过期 quote 不能继续使用，否则"用户以为 0.1 USDC，实际执行时变成 1 USDC"。

Quote 应该能被签名或验证来源，否则 dispute 时难以证明当时接受的是什么条件。

### ⭐⭐ Payment Intent（付款意图）
表达"用户或 Agent 想为某个具体服务付款"，但不等同于已经结算。

**Payment Intent ≠ 交易已发生。** 它是"用户授权这类付款"，在真正结算前创建，给后续 payment、escrow 和 receipt 提供上下文。

至少应包含：用户目标、服务方、最大金额、币种、链、过期时间、quote 引用、是否允许自动重试、是否需要人工确认。

### ⭐⭐ x402（HTTP 402 支付）
把 HTTP 402 Payment Required 变成互联网原生支付流程。

**流程：** 客户端请求资源 → 服务返回付款要求 → 客户端完成支付 → 带付款证明重新请求。

**对 Agent 的意义：** 工具可以像处理 401 登录一样处理 402 付款 —— 先读取付款要求，检查预算，完成付款，重放请求。

**局限：** x402 只解决付款协议的一部分，不自动解决服务质量、退款、交付争议和长期订阅。

### ⭐⭐ MPP（Machine Payments Protocol）
机器与机器之间的支付协商和结算协议。

**协议化的价值：** 发现服务 → 获取报价 → 授权支付 → 结算 → 返回收据，全部机器可读。

**不需要全部上链：** 很多步骤可以链下完成，链上只承担最终结算、抵押、收据锚定或争议证据。

### ⭐⭐ Subscription（订阅）
持续服务的支付模型（API 包、持续监控、长期任务）。

**关键要求：** 用户能看到当前授权、下次扣款时间、剩余额度、服务范围，并能随时停止。

**警惕"静默续费"：** Agent 订阅数据源时，系统必须让用户知道续费时间、最高月预算、服务方是否可涨价。

**与智能账户 policy 结合：** 每月上限、服务方白名单、扣款时间窗口、异常扣款告警。不要用无限 allowance 实现订阅。

### ⭐⭐⭐ Micropayment（微支付）
高频、小额、自动化服务的支付。

**核心问题：** 每次调用都上链结算，成本可能超过服务本身。

**解决方案：** L2、payment channel、批量结算、预付余额、链下账本 + 链上最终结算。

**设计要先算经济账：** 单次服务价值、链上手续费、失败率、欺诈成本、对账成本。

**对 AI 工具尤其如此：** 一次搜索/轻量推理/网页抓取可能只有几厘，适合批量结算；高价值报告/审计/交易执行，适合 escrow 和单独 receipt。

## 在 AI × Web3 中的位置

Machine Payment 是 Agentic Commerce 的基础。Agent 要购买数据、委托另一个 Agent、调用付费 API 或为用户完成任务，都需要支付能力。

**完整链路：**
```
用户授权预算
    → Agent 获取 quote
    → 系统检查 policy
    → 付款进入 escrow 或直接结算
    → 服务交付
    → 收据留下证据
```

## 我的判断

1. **x402 是最有想象力的。** 把付款变成 HTTP 原生语义，Agent 调用付费 API 就像调用免费 API 一样自然，只是多了一步 402 处理。

2. **Budget 多层设计是被忽视的。** 大多数 Agent demo 只有"能不能付款"，没有"在什么范围内付款"。全局预算 + 任务预算 + 单次上限 + 服务方白名单才是生产级设计。

3. **Micropayment 的经济账很关键。** 不是所有服务都适合链上结算。一次几厘的 API 调用，用批量结算或预付余额比每笔上链合理得多。

4. **与 DeFi Swap Guardian 的关联：** 受限助手设计里的 Policy 规则（单笔 ≤ 1 ETH，日限 ≤ 3 ETH）就是 Budget 的具体实现。Machine Payment 把这个思路推广到了所有机器间支付场景。
