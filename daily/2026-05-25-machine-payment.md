## 一句话总结

**Machine Payment 的核心，是把"付款意图"和"实际结算"拆开，并让每一步都有凭证。**

---

## 为什么要学这个

如果 Agent 只能免费调用工具，它的能力会停在 demo 阶段。真实服务需要付费：
- 模型推理
- 数据 API
- 浏览器环境
- 链上交易
- 存储
- 人工审核
- 另一个 Agent 的任务执行

**核心挑战不是转账本身，而是：**
1. 谁授权？
2. 价格是多少？
3. 什么时候扣款？
4. 失败怎么退？
5. 收据如何验证？
6. 预算如何不被滥用？

---

## 第一性原理

> Agent 不应该拥有无限支付能力，只应该拿到**具体任务、预算和收款方范围内的支付权限**。

三条基本原则：
1. **预算先于执行** — 没有预算边界，就没有安全自动支付
2. **报价必须可比较** — Agent 要知道价格、币种、有效期、服务范围和退款条件
3. **收据必须可验证** — 付款后要能证明付给谁、为什么付、交付了什么

## 8 个知识节点（深度版）

### 1. ⭐ Stablecoin Payment（稳定币支付）

**定义：** 稳定币适合机器支付：价格稳定、结算快、可编程、易验证。

**Agent 必须区分的三件事：**
- **计价币种 vs 结算币种** — USD 报价，USDC 支付
- **token address、decimals、chain id** — 不能只看到"0.1"
- **approve 本身是高风险动作** — 不能和普通付款混在一起

**判断稳定币支付方案的四个维度：**
1. 付款 token 在目标链上有足够流动性
2. 收款方接受该 token
3. 支付失败是否会消耗 gas
4. 用户是否需要提前 approve

**深入理解：**
- 稳定币作为计价单位，但仍要处理 chain id、token address、decimals、allowance、gas 和跨链可用性
- 可能通过 Paymaster 或中间路由把其他 token 转成稳定币
- Agent 不能只看到"0.1"，必须知道这是 0.1 USDC、0.1 USDT 还是 0.1 原生 gas token

---

### 2. ⭐ Budget（预算）

**定义：** 用户授权给 Agent 的最大支出范围。

**预算应拆成多层：**
| 层级 | 说明 | 示例 |
|------|------|------|
| 全局预算 | 总支出上限 | 每月最多 100 USDC |
| 任务预算 | 单个任务上限 | 本次搜索任务最多 2 USDC |
| 单次调用上限 | 单次 API 调用 | 单次 ≤ 0.5 USDC |
| 服务方上限 | 特定服务方 | 对服务商 A 最多 10 USDC/月 |
| 紧急停止条件 | 异常检测 | 连续 3 次失败则停止 |

**常见错误：** 只设总预算，不设频率和服务方范围 → Agent 可以把全部预算花在一个异常报价上

**预算系统应能回答：**
- 这笔付款是否属于当前任务？
- 是否在授权服务范围内？
- 是否超过频率或金额限制？

**深入理解：**
- 预算不应该只存在聊天里，而要进入 wallet policy、session key 或后端账本
- "今天最多花 5 USDC 调用数据 API"、"单次浏览器任务不超过 0.2 USDC"
- Agent 即使被诱导反复调用工具，也只能消耗很小范围内的额度

---

### 3. ⭐⭐ Quote（报价）

**定义：** 服务方给 Agent 的可执行报价。

**合格 quote 至少包含：**
- 服务内容
- 价格
- 币种
- 收款地址
- 有效期
- 交付条件
- 退款条件
- quote id

**有效期的重要性：**
- 价格、汇率、服务容量和链上 gas 都会变化
- 过期 quote 不能继续使用
- 否则"用户以为 0.1 USDC，实际执行时变成 1 USDC"

**Quote 验证：**
- 应该能被签名或验证来源
- 没有签名、没有 quote id、没有收款地址绑定 → dispute 时难以证明当时接受的条件

---

### 4. ⭐⭐ Payment Intent（付款意图）

**定义：** 表达"用户或 Agent 想为某个具体服务付款"，但**不等于已经结算**。

**Payment Intent ≠ 交易已发生。** 它是：
- "用户授权这类付款"
- 在真正结算前创建
- 给后续 payment、escrow 和 receipt 提供上下文

**至少应包含：**
- 用户目标
- 服务方
- 最大金额
- 币种
- 链
- 过期时间
- quote 引用
- 是否允许自动重试
- 是否需要人工确认

**深入理解：**
- Payment Intent 应该绑定任务、金额、收款方、有效期和可接受结果
- 即使后续由 Agent 自动执行，也能回到用户的原始授权
- 缺少这些字段，Agent 的付款行为就很难审计

---

### 5. ⭐⭐ x402（HTTP 402 支付）— **核心节点**

**定义：** 把 HTTP 402 Payment Required 变成互联网原生支付流程。

**标准流程：**
```
客户端请求资源
  → 服务返回 402 + 付款要求（金额、币种、收款地址）
  → 客户端完成支付
  → 带付款证明重新请求
  → 服务返回资源
```

**对 Agent 的意义：**
- 工具可以像处理 401 登录一样处理 402 付款
- 先读取付款要求 → 检查预算 → 完成付款 → 重放请求
- 把"付费才能访问资源"放回 HTTP 语义里

**x402 的价值：**
- API、数据、内容和 agent service 的小额按次付费
- 标准化了"付费才能访问"的协议层
- Agent 调用付费 API 就像调用免费 API 一样自然

**x402 的局限：**
- 只解决付款协议的一部分
- 不自动解决服务质量、退款、交付争议和长期订阅
- 高价值服务仍然需要 escrow、receipt 和 dispute 机制

**与 CAW 的关系：**
- x402 是协议层（HTTP 语义）
- CAW（Chain-Aware Wallet）是执行层（链上支付）
- 两者结合形成完整的支付闭环：x402 定义"付多少、付给谁"，CAW 负责"怎么付"

---

### 6. ⭐⭐ MPP（Machine Payments Protocol）

**定义：** 机器与机器之间的支付协商和结算协议。

**协议化的价值：**
```
发现服务 → 获取报价 → 授权支付 → 结算 → 返回收据
```
全部机器可读。

**关键理解：**
- 不需要全部上链
- 很多步骤可以链下完成
- 链上只承担：最终结算、抵押、收据锚定、争议证据
- 如果没有协议，Agent 只能读网页或 API 文档临时拼流程，可靠性很差

---

### 7. ⭐⭐ Subscription（订阅）

**定义：** 持续服务的支付模型（API 包、持续监控、长期任务）。

**关键要求：**
- 用户能看到当前授权
- 下次扣款时间
- 剩余额度
- 服务范围
- 能随时停止

**警惕"静默续费"：**
- Agent 订阅数据源时，系统必须让用户知道：
  - 续费时间
  - 最高月预算
  - 服务方是否可涨价
  - 失败时是否继续扣款

**与智能账户 policy 结合：**
- 每月上限
- 服务方白名单
- 扣款时间窗口
- 异常扣款告警
- **不要用无限 allowance 实现订阅**

---

### 8. ⭐⭐⭐ Micropayment（微支付）

**定义：** 高频、小额、自动化服务的支付。

**核心问题：** 每次调用都上链结算，成本可能超过服务本身。

**解决方案：**
- L2
- Payment channel
- 批量结算
- 预付余额
- 链下账本 + 链上最终结算

**设计要先算经济账：**
- 单次服务价值
- 链上手续费
- 失败率
- 欺诈成本
- 对账成本

**对 AI 工具尤其如此：**
- 一次搜索/轻量推理/网页抓取 → 可能只有几厘 → 适合批量结算
- 高价值报告/审计/交易执行 → 适合 escrow 和单独 receipt

---

## 最小实践：Agent 购买 API 的支付流程

```
1. 用户授权：今天最多 3 USDC
2. API 返回 quote：一次调用 0.1 USDC，5 分钟内有效
3. Agent 检查预算和服务方身份
4. 钱包或支付工具完成付款
5. API 返回结果和 receipt
6. 系统记录 quote、payment intent、交易哈希、结果和剩余预算
```

## 在 AI × Web3 中的位置

Machine Payment 是 **Agentic Commerce 的基础**。

**完整链路：**
```
用户授权预算
  → Agent 获取 quote
  → 系统检查 policy
  → 付款进入 escrow 或直接结算
  → 服务交付
  → 收据留下证据
```

**与前几章的关联：**

| 章节 | 关系 |
|------|------|
| Agent Wallet | 提供支付权限边界（Session Key、Policy） |
| Agent Workflow | 支付是 workflow 里的一个步骤 |
| Agent Identity | 服务方身份验证（quote 签名、收款方白名单） |
| Settlement & Escrow | 支付的后续章节，处理争议和结算 |

## 我的判断（深度版）

1. **x402 是最有想象力的。** 把付款变成 HTTP 原生语义，Agent 调用付费 API 就像调用免费 API 一样自然，只是多了一步 402 处理。这是互联网支付的范式转变。

2. **Budget 多层设计是被忽视的。** 大多数 Agent demo 只有"能不能付款"，没有"在什么范围内付款"。全局预算 + 任务预算 + 单次上限 + 服务方白名单才是生产级设计。

3. **Micropayment 的经济账很关键。** 不是所有服务都适合链上结算。一次几厘的 API 调用，用批量结算或预付余额比每笔上链合理得多。

4. **与 DeFi Swap Guardian 的关联：** 受限助手设计里的 Policy 规则（单笔 ≤ 1 ETH，日限 ≤ 3 ETH）就是 Budget 的具体实现。Machine Payment 把这个思路推广到了所有机器间支付场景。

5. **x402 + CAW = 完整支付闭环：** x402 定义协议层（HTTP 语义），CAW 提供执行层（链上支付）。两者结合才能让 Agent 在互联网上无缝付费。
