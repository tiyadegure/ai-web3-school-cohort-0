## 📖 今日自学

### 1. Agent Workflow 章节（35 分钟）

**核心定义：** 把概率模型放进确定性流程里

**7 个知识节点：**

**⭐⭐ Task Graph（任务图）**
把目标拆成节点和依赖，每步设置输入/输出/权限/停止条件。例：一笔 swap 拆成 9 步（读取目标 → 查余额 → 查价格 → 生成交易 → 模拟 → 展示风险 → 用户确认 → 发送 → 追踪）

**⭐⭐⭐ State Machine（状态机）**
链上工作流状态：draft → context_loaded → plan_ready → waiting_user_confirmation → submitted → confirmed / reverted。价值：系统不会忘记自己在哪，不会重复执行危险动作

**⭐⭐ Human-in-the-loop（重点）**
不是所有操作都需要人确认，按风险分层：
- L1 只读分析 → 自动执行
- L2 交易草稿 → 自动生成
- L3 小额白名单 → session key 执行
- L4 高风险交易 → 必须人工确认
- L5 超出 policy → 直接拒绝

**关键洞察：** 重点不是"有没有人工确认"，而是人确认时能否看懂资产变化和失败风险

**⭐⭐ Retry / Fallback**
Web3 不能盲目重试：交易 pending 不能简单再发一笔；RPC 异常可切换 provider 但要记录来源

**⭐ Trace**
每一步的输入/判断/工具调用/结果记录。有 trace 才能复盘是模型错、工具错还是用户确认了错误动作

**⭐⭐⭐ Evaluation Harness + Regression Set**
用固定测试用例防止安全退化：正常 swap、错误链请求、无限 approve、恶意文档诱导、余额不足、预言机价格过旧...

### 2. Agent Wallet 章节（40 分钟）

**核心定义：** Agent Wallet 不是"给 AI 一个私钥"，而是让用户把一小部分**可撤销、可审计**的权限交给 Agent。

**一句话：** 控制权不能交给一个概率系统。Agent 只能拿到可验证、可限制、可撤销的行动空间。

#### EOA vs Smart Account vs Multisig 对比

| 维度 | EOA | Smart Account | Multisig/Safe |
|------|-----|---------------|---------------|
| 控制方式 | 一把私钥 | 合约规则（可编程） | 多人签名 N-of-M |
| 适合 Agent？ | ❌ 全有或全无 | ✅ 最适合 — 规则内执行 | ✅ Agent 生成草稿，人多签 |
| Session Key | 不支持 | 原生支持 | 通过模块支持 |
| 撤销能力 | 只能转走资产 | 可编程撤销 | 可轮换签名者 |
| Gas 支付 | 只能用 ETH | Paymaster 可代付 | 通常用 ETH |

**关键洞察：** EOA 的问题是"全有或全无"；Smart Account 让账户本身拥有规则；Safe 让执行权拆开。

#### Session Key 权限边界

Session Key = 临时、有限权限的小钥匙。不是小号私钥，而是一组受限能力。

合格的 Session Key 至少限制：
1. 有效时间
2. 可调用的合约/方法
3. 单笔金额和总额度
4. 可操作的 token 类型
5. 是否允许转出外部地址
6. 用户随时可撤销

#### Agent 不接触私钥的 3 层保障

```
用户主资产（主私钥）
    │ 授权一小部分权限
    ▼
Smart Account（合约规则层）
    │ 分配临时受限权限
    ▼
Session Key（Agent 持有）
    │ 只能在规则内执行
    ▼
Agent 执行低风险操作
```

- **第 1 层：Smart Account 规则** — 超出规则直接 revert
- **第 2 层：Guard 拦截** — 交易发出前确定性规则检查
- **第 3 层：Simulation 预演** — 签名前先模拟结果

#### 其他节点

**⭐⭐⭐ Policy（策略）** — 把"Agent 可以做什么"写成系统能检查的规则
**⭐⭐⭐ Guard（拦截）** — 确定性拦截层，Agent 负责生成，Guard 负责拒绝
**⭐⭐ Simulation（模拟）** — 签名前预演，回答"这笔交易会让用户失去什么、得到什么、承担什么风险"
**⭐⭐ Revocation（撤销）** — 所有自动化权限都应有用户可见的关闭入口

### 产出 2：受限 Web3 助手设计 — DeFi Swap Guardian（WCB 40 分）

**解决的问题：** Agent 要么不能执行（废物），要么拿到全部权限（危险），需要找中间路线。

**设计概要：**
- **8 步 Task Graph：** 意图解析 → 上下文加载 → 价格查询 → 交易构建 → 模拟 → Guard 检查 → 摘要 → 等待确认
- **10 个状态的状态机：** idle → context_loading → planning → simulating → guard_checking → awaiting_human → signing → submitting → confirming → completed
- **5 个人工确认点：** 摘要确认 / 首次授权 / 高金额 / 钱包签名 / 异常恢复
- **6 个风险：** MEV / 模拟不一致 / Prompt Injection / RPC 不可信 / Session Key 泄露 / Gas 飙升
- **Policy 规则：** 单笔 ≤ 1 ETH，日限 ≤ 3 ETH，仅白名单合约，禁止无限 approve

**核心洞察：** Agent 负责生成，Guard 负责拒绝。确定性规则比概率模型更可靠。

### 产出 3：EOA vs 智能账户 vs 多签权限对比（WCB 30 分）

**从安全性、灵活性、成本三个维度对比三种账户在 Agent 场景下的表现：**

**安全性排序：** 多签 > 智能账户 >> EOA（Agent 场景下）

**核心结论：**
- **EOA** → 一把钥匙开一扇门，给了 Agent 钥匙就等于给了全部
- **智能账户** → 一套门禁系统，可以给 Agent 一张限时/限额/限区域的门禁卡
- **多签** → 银行金库，Agent 可以提议开库，但需要多人到场

**推荐方案：**
- 个人 + Agent → 智能账户 + Session Key
- 团队/DAO + Agent → Safe 多签 + Agent 模块
- 临时操作 → EOA（Agent 不接触私钥）
