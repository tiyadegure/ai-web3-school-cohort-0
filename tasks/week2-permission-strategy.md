## 二、权限策略设计

### 2.1 预算控制

```
┌─────────────────────────────────────────────────┐
│                   预算分层                        │
├─────────────────────────────────────────────────┤
│  全局预算上限    │ 单 Agent 总限额：100 USDC/月  │
│  任务预算上限    │ 单次任务：10 USDC             │
│  单笔限额       │ 单笔交易：5 USDC              │
│  服务预算       │ 单个服务方：20 USDC/月         │
│  紧急停止       │ 一键冻结所有支出               │
└─────────────────────────────────────────────────┘
```

**设计原则：**
- 不只设总预算——每层都有独立上限
- 超额需要人工确认
- 紧急停止按钮在首页可见

### 2.2 可调用合约白名单

| 合约 | 操作 | 风险等级 | 限制 |
|------|------|----------|------|
| Aave V3 Pool | `supply` / `withdraw` | 🟡 中 | 单笔 ≤ 10 USDC |
| Uniswap V3 Router | `exactInputSingle` | 🟡 中 | 滑点 ≤ 1% |
| ERC-20 各 Token | `balanceOf` / `allowance` | 🟢 低 | 只读 |
| 任意合约 | `eth_call` | 🟢 低 | 只读 |
| 任意合约 | `approve` | 🔴 高 | 需人工确认 |
| 任意合约 | `transfer` | 🔴 高 | 需人工确认 + 多签 |

**不在白名单上的合约调用 → 直接拒绝**

### 2.3 可执行动作分类

```
🟢 低风险（自动执行）
├── eth_call（只读查询）
├── 余额查询
├── 交易历史查询
├── 合约状态查询
└── Gas 估算

🟡 中风险（自动 + 日志）
├── DEX Swap（白名单合约，限额内）
├── Lending supply/withdraw（白名单合约，限额内）
└── DCA 策略执行（预设参数）

🔴 高风险（人工确认）
├── approve / setApprovalForAll
├── 转账 > 单笔限额
├── 新合约首次交互
├── 合约升级/代理操作
└── 多签交易发起
```

### 2.4 人工确认阈值

| 触发条件 | 阈值 | 确认方式 |
|----------|------|----------|
| 单笔金额 | > 5 USDC | 手机推送确认 |
| 24h 累计 | > 20 USDC | 手机推送确认 |
| 新合约首次交互 | 任意金额 | 手机推送确认 |
| approve 操作 | 任意金额 | 手机推送 + 二次确认 |
| 非白名单合约 | 任意金额 | 直接拒绝 |
| Gas 价格 | > 100 gwei | 手机推送确认 |
| 滑点 | > 2% | 拒绝 |

### 2.5 撤销方式

```
撤销层级：
├── Level 1: 撤销单个 Pact（任务级）
│   └── 用户在 Cobo App 中撤销当前活跃 Pact
├── Level 2: 撤销 Session Key（Agent 级）
│   └── 一键撤销某个 Agent 的所有 Session Key
├── Level 3: 紧急停止（全局）
│   └── 一键冻结所有 Agent 的所有操作
└── Level 4: 链上撤销（合约级）
    └── 调用 token.approve(address, 0) 撤销授权
```

**关键设计：**
- Level 1-3 不需要链上交易（即时生效）
- Level 4 需要链上交易（需要 Gas）
- 紧急停止按钮在首页、通知栏、Cobo App 都可触达

### 2.7 失败处理

| 失败类型 | 处理方式 | 用户通知 |
|----------|----------|----------|
| Session Key 过期 | 自动提示续期 | 推送通知 |
| Pact 余额不足 | 暂停 + 提示充值 | 推送通知 |
| 交易 revert | 重试 1 次 → 失败退款 | 推送通知 |
| Gas 不足 | 自动提高 Gas → 仍失败则暂停 | 推送通知 |
| 越权尝试 | 自动撤销 Session Key + 告警 | 紧急通知 |
| 白名单外合约 | 直接拒绝 | 推送通知 |

## 三、ERC-4337、Safe、Guard/Policy 机制为什么重要

### ERC-4337（Account Abstraction）

**解决的问题：** EOA 钱包权限是二元的——有私钥就有完全控制权，没有就没有。

**为什么重要：**
- 把钱包从"有私钥就能做任何事"变成"可编程的权限规则"
- Agent 可以有自己的 Smart Account，权限由 Policy 定义而不是私钥
- 支持批量操作、Gas 代付、社交恢复

**在 Agent 场景中：**
- Agent 的 Smart Account 可以设置"只允许调用白名单合约"
- Session Key 通过 ERC-4337 的 validation 逻辑实现细粒度权限
- 用户不需要把 EOA 私钥给 Agent

### Safe（Multi-sig Smart Account）

**解决的问题：** 单一签名者 = 单点故障。

**为什么重要：**
- 多签要求多个签名才能执行交易
- Agent + 用户 + Cobo 基础设施 = 三方 MPC 签名
- 即使 Agent 被攻破，也无法单独签名

**在 Agent 场景中：**
- CAW 的 MPC 模式本质就是 Safe 的变体
- 高价值操作需要 2/3 签名
- 日常低风险操作可以 1/3（Agent 独立执行）

### Guard / Policy 机制

**解决的问题：** 合约层面的权限约束，而不是应用层面。

**为什么重要：**
- Guard 在合约执行前检查交易是否符合规则
- Policy 定义"什么可以做，什么不能做"
- 在基础设施层强制执行，不是靠 Agent 自律

**在 Agent 场景中：**
- Safe Guard：检查交易目标是否在白名单、金额是否超限、是否需要人工确认
- Cobo Pact：按任务动态生成权限边界，完成后自动过期
- 即使 LLM 被 prompt injection 诱导生成恶意交易，Guard 也会拦截

### 三者组合

```
ERC-4337（可编程账户）
      +
Safe（多签保护）
      +
Guard/Policy（合约层权限）
      ↓
Agent 钱包的三层防护：
  1. 账户层：权限可编程（ERC-4337）
  2. 签名层：多签防单点故障（Safe/MPC）
  3. 执行层：合约层强制规则（Guard/Policy）
```

## 四、与 Threat Model 的关系

本权限策略是 Threat Model 的具体落地：

| Threat Model 威胁 | 权限策略对应 |
|-------------------|-------------|
| T1: Prompt Injection | 工具白名单 + 非白名单合约直接拒绝 |
| T2: 工具返回伪造 | 交易模拟 + 结果验证 |
| T3: 越权操作 | Session Key 最小权限 + 越权自动撤销 |
| T4: 数据泄露 | 只读工具自动通过，写入工具需确认 |
| T5: 模型幻觉 | 人工确认阈值 + 置信度检查 |
| T6: Pact 绕过 | 预算分层 + Guard 合约层拦截 |

## 五、与主方向 Privacy 的结合

权限策略本身就是隐私保护的一部分：

- **最小权限 = 最小数据暴露**：Agent 只能访问完成任务所需的数据
- **Session Key = 临时身份**：不需要暴露主私钥
- **MPC 签名 = 密钥分片**：私钥不完整暴露给任何一方
- **Audit Log = 可验证但不泄露**：记录操作但不记录原始数据

**核心洞察：** 最安全的工具不是功能最多，而是刚好能完成任务且无法越界。权限策略 = Privacy + Security 的工程实现。
