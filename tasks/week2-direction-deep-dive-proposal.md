## 二、主方向选择说明：Privacy / Security / Sovereignty

### 为什么不是纯 AI 问题？

- AI 模型本身不提供隐私保证——LLM 推理过程对用户不可见，prompt 数据可能被记录
- AI 无法自证"我没有泄露你的数据"——需要密码学证明（ZK）或硬件保证（TEE）
- AI 的安全审计需要可信执行环境，纯软件层面无法防篡改
- 数据主权不能靠 AI 的"承诺"，需要链上权限控制和加密机制

### 为什么不是纯 Web3 问题？

- 链上数据默认公开——需要 AI 判断哪些数据该公开、哪些该隐藏
- ZK 电路的设计和验证需要 AI 辅助——人工编写 ZK 电路极其复杂
- 隐私策略需要语义理解——"这条交易记录是否包含敏感信息"是 AI 判断问题
- 恶意行为检测需要模式识别——纯规则引擎无法覆盖 AI Agent 的攻击变体

### AI × Web3 交叉点

| 交叉 | 说明 |
|------|------|
| ZK + AI | 用零知识证明验证 AI 推理正确性，不暴露模型参数和用户输入 |
| TEE + Agent | 在可信执行环境中运行 Agent，保证代码完整性 |
| 匿名身份 + AI 风控 | Semaphore 等匿名协议 + AI 行为分析 = "匿名但可问责" |

## 三、问题拆解：隐私 Agent 支付场景

### 参与方

| 参与方 | 角色 | 隐私需求 |
|--------|------|----------|
| 用户 | 授权预算、设定隐私偏好 | 交易金额、交易对手方、数据查询意图 |
| AI Agent | 执行任务、比较报价、发起支付 | 用户 prompt、决策过程、中间状态 |
| 服务方（API） | 提供数据/服务 | 定价策略、用户画像 |
| 区块链网络 | 结算和记录 | 交易详情需要选择性公开 |
| ZK/TEE 基础设施 | 提供隐私保证 | 执行证明、数据加密 |

```
└─────────────────────────────────────────────────────────────┘
  │
  ▼

└─────────────────────────────────────────────────────────────┘

  ▼

│ Step 3: Agent 发起支付（隐私边界）                            │

│ - ⚠️ 风险：支付交易暴露金额和接收方                           │
│ - 解决方案：通过 Railgun 隐私支付，或使用 Payment Channel      │
│ - CAW 执行：签名交易、提交到 Facilitator                     │
└─────────────────────────────────────────────────────────────┘
  │
  ▼

│ Step 4: Agent 获取数据（隐私边界）                            │
│ - API 返回 DeFi 持仓数据                                     │
│ - ⚠️ 风险：数据可能包含其他用户信息                           │
│ - 解决方案：ZK 证明数据正确性，不暴露原始数据                  │
└─────────────────────────────────────────────────────────────┘
  │
  ▼
┌─────────────────────────────────────────────────────────────┐
│ Step 5: Agent 返回结果（隐私边界）                            │
│ - 只返回用户关心的汇总信息                                    │
│ - 不暴露：查询过程、支付详情、中间状态                         │
│ - 链上记录：只记录交易哈希，不记录查询内容                     │
└─────────────────────────────────────────────────────────────┘
```

### AI 作用 vs Web3 机制

| 环节 | AI 作用 | Web3 机制 |
|------|---------|-----------|
| 意图解析 | 理解用户自然语言指令 | — |
| 隐私判断 | 判断数据敏感级别 | — |
| 服务发现 | 搜索和比较 API | 链上服务注册（ERC-8004） |
| 报价比较 | 评估性价比 | x402 Quote |
| 支付决策 | 检查预算和风险 | Session Key + Policy |
| 支付执行 | — | Railgun / Payment Channel |
| 数据验证 | — | ZK Proof |
| 结果摘要 | 提取关键信息 | 链上收据（哈希） |

### 自动化边界

| 可自动 | 需人工确认 |
|--------|-----------|
| 意图解析 | 首次隐私偏好设置 |
| 服务搜索 | 超出预算的支付 |
| 报价比较 | 新服务方授权 |
| 预算内支付 | 隐私级别变更 |
| 数据格式化 | 异常交易告警 |

### 验证方式

| 验证项 | 方法 |
|--------|------|
| 支付正确性 | 链上交易记录 + x402 收据 |
| 数据完整性 | ZK Proof（证明数据来自可信源） |
| 执行正确性 | TEE Attestation（证明代码未被篡改） |
| 隐私保护 | 链上无关联记录（Railgun） |
| 预算合规 | Session Key Policy 审计 |

### 主要风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Agent 泄露用户意图 | 隐私泄露 | ZK 查询 + Tor |
| 支付交易暴露身份 | 隐私泄露 | Railgun 隐私支付 |
| 恶意 API 返回虚假数据 | 经济损失 | ZK 数据验证 |
| Agent 被 prompt injection 操纵 | 资金风险 | Policy 限制 + 人工确认 |
| TEE 硬件漏洞 | 执行不可信 | 多层验证 + 审计 |

## 四、项目初步 Proposal：Privacy Agent Framework

### 项目名称
**ShadowAgent** — 隐私优先的 AI Agent 支付框架

### 目标用户
- 需要隐私保护的 DeFi 用户（不想暴露持仓和交易策略）
- 企业用户（不想暴露商业查询意图）
- 隐私敏感的数据消费者

### 真实场景
**场景：匿名 DeFi 投研**

用户想查询某地址的 DeFi 持仓，但不想暴露：
1. 自己在查询谁（查询意图）
2. 用了哪个 API（服务方关系）
3. 支付了多少（交易金额）

ShadowAgent 流程：
1. 用户通过加密通道发送查询请求
2. Agent 在 TEE 中解析意图，生成 ZK 查询
3. 通过 Railgun 进行隐私支付
4. API 返回数据，Agent 在 TEE 中验证
5. 只返回汇总结果，链上只记录哈希

### 最小功能（MVP）

| 功能 | 优先级 | 技术栈 |
|------|--------|--------|
| 隐私查询代理 | P0 | TEE + x402 |
| 隐私支付 | P0 | Railgun / Payment Channel |
| 预算控制 | P0 | Session Key + Policy |
| 数据验证 | P1 | ZK Proof |
| 匿名身份 | P1 | Semaphore |
| 声誉系统 | P2 | ERC-8004 |

### 验证方式

| 验证项 | 方法 | 成功标准 |
|--------|------|----------|
| 隐私保护 | 链上分析 | 无法关联用户与查询 |
| 支付正确性 | 对账 | 金额和服务匹配 |
| 数据完整性 | ZK 验证 | 数据来自可信源 |
| 预算合规 | 审计 | 不超预算 |

### 主要风险

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| TEE 硬件漏洞 | 低 | 高 | 多层验证 |
| Railgun 被制裁 | 中 | 高 | 备选方案（Payment Channel） |
| Agent 被操纵 | 中 | 中 | Policy + 人工确认 |
| 用户体验差 | 高 | 中 | 渐进式隐私（默认不开启全隐私） |

### 可能赛道

- **DeFi 隐私工具**：竞争者少，用户需求明确
- **企业数据采购**：B2B 场景，付费意愿高
- **隐私 Agent 基础设施**：平台型，可扩展

### Week 3 下一步

1. **技术验证**：搭建 TEE + x402 的最小原型
2. **用户访谈**：找 3-5 个 DeFi 用户验证需求
3. **竞品分析**：调研 Railgun、Aztec、Penumbra 的 Agent 集成方案
4. **MVP 设计**：确定第一版功能范围和技术选型

### 典型场景：匿名 DeFi 持仓查询

**用户需求：** 查询某巨鲸地址的 DeFi 持仓，不想暴露自己在关注这个地址

**传统方式的问题：**
- 直接调用 Etherscan API → 查询记录可被追踪
- 使用公共 RPC → 节点运营商知道你在查谁
- 支付 API 费用 → 交易记录暴露金额和服务方

**ShadowAgent 方式：**
1. 用户通过加密通道发送请求（Agent 不知道用户真实身份）
2. Agent 在 TEE 中解析意图，生成 ZK 查询（API 不知道查询来源）
3. 通过 Railgun 支付（链上无法关联支付与用户）
4. 数据在 TEE 中验证和处理（原始数据不离开安全环境）
5. 只返回汇总结果（"该地址在 Aave 有 100 ETH 存款"）

**隐私保证：**
- 链上无关联记录
- API 不知道查询来源
- Agent 不知道用户真实身份
- 只有用户自己知道查询内容

### 反例：隐私泄露场景

**场景：** 用户查询竞争对手的 DeFi 策略

**如果不用隐私 Agent：**
1. 用户直接调用 API → 查询记录可被竞争对手发现
2. 支付记录暴露 → 竞争对手知道有人在关注他
3. 查询模式分析 → 竞争对手推断出用户的策略意图

**后果：**
- 竞争对手提前调整策略
- 用户的交易意图被暴露
- 可能导致经济损失

**为什么需要 AI × Web3 交叉解决：**
- 纯 AI：无法保证查询不被记录
- 纯 Web3：无法理解查询语义和隐私级别
- AI × Web3：AI 判断隐私级别 + Web3 提供隐私保证

### 关键风险分析

| 风险 | 技术原因 | 影响 | 缓解方案 |
|------|----------|------|----------|
| TEE 侧信道攻击 | 硬件设计缺陷 | 隐私泄露 | 多层验证 + 审计 |
| ZK 电路 Bug | 实现错误 | 数据泄露 | 形式化验证 + 审计 |
| Railgun 被制裁 | 监管压力 | 支付不可用 | 备选方案（Payment Channel） |
| Agent 被 Prompt Injection | LLM 固有缺陷 | 资金风险 | Policy + 人工确认 |
| 用户体验差 | 隐私与便利的权衡 | 采用率低 | 渐进式隐私 |

## 八、总结

### 核心洞察

**隐私是 AI Agent 的基础设施，不是附加功能。**

从 Chain-aware Context 到 Settlement & Escrow，每一层都需要隐私保证：
- Context 层：链上数据读取可能暴露用户意图
- Wallet 层：交易签名暴露身份
- Payment 层：支付金额和接收方敏感
- Settlement 层：交付内容可能包含商业机密

### AI × Web3 交叉的独特价值

- **纯 AI 做不到：** 无法自证隐私、无法防篡改、无法保证执行正确性
- **纯 Web3 做不到：** 无法理解语义、无法判断隐私级别、无法智能决策
- **AI × Web3 能做到：** ZK 验证 AI 推理、TEE 保护 Agent 执行、匿名身份 + AI 风控

### 项目方向

**ShadowAgent** — 隐私优先的 AI Agent 支付框架
- 目标：让用户在享受 Agent 便利的同时，保持数据主权和隐私
- 技术：TEE + x402 + Railgun + ZK + Semaphore
- 场景：匿名 DeFi 投研、企业数据采购、隐私敏感的自动化任务
