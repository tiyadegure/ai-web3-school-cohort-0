## 一、场景选择：隐私数据分析 Agent

**场景描述：** 用户需要分析自己的链上交易历史（例如：过去 30 天的 DeFi 收益、Gas 消耗、风险暴露），但不想把原始交易数据暴露给任何第三方。一个 Privacy Analytics Agent 提供这项服务：在 TEE 环境中读取用户数据 → 完成分析 → 返回结果，全程数据不泄露。

**为什么选这个场景？**
- 与主方向（Privacy / Security / Sovereignty）直接相关
- 涉及敏感数据，支付流程必须与隐私保护结合
- 现实需求：链上数据分析是高频场景，但隐私保护几乎空白

## 二、完整流程拆解（8 个环节）

### 1. 谁下单（Request）

| 角色 | 说明 |
|------|------|
| **下单方** | 用户（人类或用户 Agent） |
| **动作** | 发起分析请求，指定分析范围（时间、地址、分析类型） |
| **关键信息** | 分析目标、数据范围、期望输出格式、预算上限 |

```
用户 → Agent: "分析我过去 30 天的 DeFi 收益，预算 5 USDC"
```

### 2. 谁报价（Quote）

| 角色 | 说明 |
|------|------|
| **报价方** | Privacy Analytics Agent |
| **动作** | 根据分析复杂度、数据量、TEE 计算成本报价 |
| **报价内容** | 价格（USDC）、预计耗时、输出格式、隐私保证级别 |

```
Agent → 用户: "分析 30 天 DeFi 收益，价格 3 USDC，预计 2 分钟，
               TEE 环境执行，数据不离开安全飞地"
```

**报价依据：**
- 数据量（交易笔数）→ 计算成本
- 分析复杂度（简单统计 vs 深度归因）→ 时间成本
- 隐私级别（普通 vs TEE vs ZK）→ 基础设施成本

### 3. 预算授权（Budget Authorization）

| 角色 | 说明 |
|------|------|
| **授权方** | 用户 |
| **动作** | 批准预算，设置支付条件 |
| **授权方式** | Session Key 或 PACT 预算限制 |

```
用户 → 智能合约: "授权 Agent 从我的钱包提取最多 5 USDC，
                 仅限本次分析任务，有效期 10 分钟"
```

**安全边界：**
- 金额上限：不能超过用户设定的单笔限额
- 时间窗口：Session Key 过期自动失效
- 用途限制：只能用于支付分析服务，不能转给第三方
- 撤销机制：用户可随时撤销授权

### 4. 谁执行（Execute）

| 角色 | 说明 |
|------|------|
| **执行方** | Privacy Analytics Agent |
| **执行环境** | TEE（可信执行环境）|
| **执行过程** | 读取用户链上数据 → 分析计算 → 生成结果 → 生成执行证明 |

```
Agent (TEE 内部):
  1. 读取用户地址的交易历史（通过 RPC/Explorer API）
  2. 分类统计：收益、Gas、风险暴露
  3. 生成分析报告
  4. 输出 TEE Attestation（证明代码完整性 + 数据未泄露）
```

**隐私保证：**
- 数据在 TEE 飞地内处理，外部无法读取
- 执行代码哈希可验证（Attestation）
- 数据处理完毕后立即销毁，不留副本

### 5. 谁验收（Acceptance）

| 角色 | 说明 |
|------|------|
| **验收方** | 用户 + 自动化检查 |
| **验收条件** | 结果完整、格式正确、隐私证明有效、无数据泄露 |

**验收清单：**
- ✅ 结果包含所有要求的分析维度
- ✅ 输出格式符合约定（JSON/Markdown）
- ✅ TEE Attestation 有效（证明在安全环境中执行）
- ✅ 无原始数据泄露（结果中不包含完整交易记录）
- ✅ 执行时间在预期范围内

**验收方式：**
- 自动检查：验证 TEE Attestation、检查输出格式
- 用户确认：查看分析结果是否满足需求
- Challenge Window：用户有 5 分钟提出异议

### 6. 谁付款 / 退款 / 争议（Settlement）

#### 付款
```
智能合约: 从用户钱包 → Agent 钱包，转移 3 USDC
前提: 验收通过 + 无争议
```

#### 退款
```
触发条件:
- Agent 执行超时（> 5 分钟）
- TEE Attestation 验证失败
- 结果不完整或格式错误
退款金额: 全额退款（3 USDC）
退款路径: 智能合约自动退款 → 用户钱包
```

#### 争议
```
触发条件: 用户在 Challenge Window 内提出异议
争议内容: "结果不准确" 或 "怀疑数据泄露"
处理方式:
  1. 冻结资金（不释放给 Agent）
  2. 提交争议证据（TEE Attestation + 执行日志）
  3. 仲裁者评估（人工或去中心化仲裁）
  4. 裁决：释放给 Agent 或退款给用户
```

### 7. 记录证明（Receipt）

| 证明类型 | 内容 |
|----------|------|
| **支付凭证** | 交易哈希、金额、时间、双方地址 |
| **执行证明** | TEE Attestation、代码哈希、执行时间 |
| **交付证明** | 分析结果哈希、输出格式验证 |
| **隐私证明** | TEE 环境证明、数据销毁证明 |

**记录用途：**
- 用户：证明支付了什么、得到了什么
- Agent：证明完成了服务、建立了声誉
- 仲裁：争议时的证据链
- 声誉系统：ERC-8004 Reputation Registry 的输入

### 8. 谁仲裁（Dispute）

| 角色 | 说明 |
|------|------|
| **仲裁者** | 去中心化仲裁协议（如 Kleros）或人工审核 |
| **仲裁成本** | 申请人支付仲裁费（防止滥用） |
| **仲裁依据** | TEE Attestation + 执行日志 + 交付证明 |

**仲裁流程：**
```
1. 用户提交争议 + 仲裁费
2. 冻结资金
3. 仲裁者审查证据：
   - TEE Attestation 是否有效？
   - 执行代码是否匹配声明？
   - 结果是否完整？
4. 裁决：
   - Agent 无过错 → 释放资金给 Agent + 退还仲裁费
   - Agent 有过错 → 退款给用户 + 扣除 Agent 声誉
```

## 四、协议对比加分：x402 vs ERC-8183

### x402 解决哪一段？

**解决：支付层（Paywall + Payment）**

- **机制：** HTTP 402 Payment Required → Agent 识别支付要求 → 自动完成支付 → 获取资源
- **解决的问题：** "Agent 如何为 API 调用付费"
- **适用场景：** 付费 API、付费数据源、付费推理服务
- **在本流程中的位置：** 如果分析服务通过 HTTP API 提供，x402 负责"Agent 识别付款要求并完成支付"这一步
- **局限：** 不解决结算、验收、争议——只管"付钱拿资源"

### ERC-8183 解决哪一段？

**解决：结算层（Settlement + Dispute + Lifecycle）**

- **机制：** 定义 Task 的生命周期状态机（Created → Assigned → Delivered → Accepted → Settled / Disputed / Refunded）
- **解决的问题：** "任务从创建到结算的完整状态流转，以及争议如何处理"
- **适用场景：** 多步骤任务、需要验收和争议机制的场景
- **在本流程中的位置：** 负责步骤 5-8（验收、付款/退款/争议、记录证明）的状态管理
- **局限：** 不解决支付方式（用什么币付）和身份验证（谁在执行）

### 两者互补关系

```
x402                    ERC-8183
  │                        │
  │  "怎么付钱"             │  "任务怎么流转"
  │  "付多少"               │  "验收怎么判定"
  │  "付给谁"               │  "争议怎么仲裁"
  │                        │
  ▼                        ▼
支付层 ←──────────────→ 结算层
```

**组合使用：** x402 负责"Agent 自动识别并完成支付"，ERC-8183 负责"任务状态流转和争议解决"。在本场景中，用户通过 x402 授权预算，Agent 通过 ERC-8183 管理任务从创建到结算的完整生命周期。

## 五、与主方向（Privacy）的结合

本流程的核心创新点是将**隐私保护嵌入支付流程的每一个环节：**

1. **报价阶段：** Agent 声明隐私级别（TEE/ZK/普通），不同级别不同价格
2. **执行阶段：** TEE 飞地保证数据不泄露
3. **验收阶段：** TEE Attestation 作为隐私证明
4. **争议阶段：** Attestation 是仲裁的关键证据
5. **记录阶段：** 隐私证明永久记录，建立 Agent 的隐私声誉

**核心洞察：** 隐私不是支付流程的"附加功能"，而是决定用户是否愿意付费的关键因素。没有隐私保证，用户不会把敏感数据交给 Agent 分析。Privacy + Payment 的结合，才是 Agent 商业化的真正门槛。
