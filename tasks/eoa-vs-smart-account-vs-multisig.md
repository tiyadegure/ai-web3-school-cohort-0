# EOA vs 智能账户 vs 多签：Agent 场景下的权限对比

## 背景

当 AI Agent 需要代表用户执行链上操作时，选择哪种账户方案直接决定了：
- Agent 能做什么、不能做什么
- 出错时损失有多大
- 用户能不能及时止损

本文从 **安全性、灵活性、成本** 三个维度对比三种方案在 Agent 场景下的表现。

---

## 一、三种账户简介

### EOA（Externally Owned Account）
传统钱包，一把私钥控制一个地址。MetaMask、Trust Wallet 默认创建的就是 EOA。

### 智能账户（Smart Account / Account Abstraction）
基于 ERC-4337 的合约账户。账户本身是智能合约，可以把签名、权限、限额、恢复等逻辑写进账户规则。

### 多签（Multisig / Safe）
需要 N 个人中的 M 个签名才能执行交易。Safe 是最常用的多签基础设施。

---

## 二、核心对比

### 权限结构

**EOA**
- 控制权：一把私钥 = 全部权限
- 发起交易：私钥持有者（用户或 Agent）
- 人工确认：无内置机制，只能靠 Agent 自觉
- 恢复：私钥丢了就没了

**智能账户**
- 控制权：合约规则，可编程
- 发起交易：规则允许的任何角色（包括 Session Key 持有者）
- 人工确认：可在合约里写死"大额需确认"
- 恢复：支持社交恢复、时间锁
- 特有能力：Session Key（临时受限权限）、Paymaster（代付 gas）、限额、白名单

**多签**
- 控制权：N-of-M 签名
- 发起交易：任何签名者可以发起，但需要 M 个确认
- 人工确认：天然多人确认
- 恢复：可轮换签名者

---

### Agent 场景适用性

| 维度 | EOA | 智能账户 | 多签 |
|------|-----|---------|------|
| **Agent 能否自动执行** | ❌ 不能 — 要么给私钥（全权），要么不能动 | ✅ 能 — Session Key 允许在限额内自动执行 | ⚠️ 部分能 — Agent 可发起，但需多人签名 |
| **权限粒度** | 无 — 全有或全无 | 细粒度 — 时间/金额/合约/方法/资产都可限制 | 中等 — 可以拆分签名者角色 |
| **出错损失上限** | 💀 全部资产 | 💰 Session Key 限额内 | 💰 需多人配合，单人无法动用 |
| **撤销速度** | 只能转走资产 | 即时（合约调用 revoke） | 轮换签名者（需多签确认） |
| **适合场景** | 个人小额、一次性操作 | Agent 自动化、频繁低风险操作 | 团队金库、DAO 资金、高价值资产 |

---

### 安全性对比

| 安全维度 | EOA | 智能账户 | 多签 |
|---------|-----|---------|------|
| **私钥泄露影响** | 💀 全部资产丢失 | 💰 仅 Session Key 限额内 | 🔒 需多个私钥同时泄露 |
| **Agent 被入侵** | 💀 全部资产 | 💰 限额 + Guard 拦截 | 🔒 需多人配合 |
| **Prompt Injection** | 💀 可能生成任意交易 | 💰 Guard 确定性拦截 | 🔒 多人审核可发现异常 |
| **合约漏洞风险** | 无（不是合约） | ⚠️ 有（合约可能有 bug） | ⚠️ 有（Safe 模块可能有 bug） |
| **单点故障** | 私钥是单点 | Session Key 可撤销，不是单点 | 多签天然无单点 |

**安全性排序：多签 > 智能账户 >> EOA**（Agent 场景下）

---

### 灵活性对比

| 灵活维度 | EOA | 智能账户 | 多签 |
|---------|-----|---------|------|
| **自动化支持** | ❌ 不支持 | ✅ Session Key + Policy | ⚠️ Agent 可发起，需人签 |
| **限额控制** | ❌ 不支持 | ✅ 原生支持 | ⚠️ 通过模块支持 |
| **Gas 支付** | 只能 ETH | Paymaster 可用任意 token | 通常 ETH |
| **批量操作** | 不支持 | 支持（multicall） | 支持（批量签名） |
| **可编程规则** | ❌ | ✅ 完全可编程 | ⚠️ 通过模块扩展 |
| **链兼容性** | 所有链 | 需要 ERC-4337 支持 | 需要 Safe 部署 |

**灵活性排序：智能账户 > 多签 > EOA**

---

### 成本对比

| 成本维度 | EOA | 智能账户 | 多签 |
|---------|-----|---------|------|
| **创建成本** | 免费 | 需要部署合约（~$2-10） | 需要部署合约（~$5-20） |
| **单笔 Gas** | 最低（基础转账） | 较高（合约逻辑执行） | 较高（多签名验证） |
| **复杂操作 Gas** | 高（每步单独交易） | 低（批量操作一次交易） | 中等 |
| **维护成本** | 无 | 模块升级成本 | 签名者管理成本 |
| **开发成本** | 无 | 需要开发 Policy/Guard | 需要配置签名者 |

**成本排序：EOA < 多签 ≤ 智能账户**（但智能账户的批量操作可节省长期 gas）

## 三、Agent 场景推荐方案

### 个人用户 + Agent 辅助
**推荐：智能账户 + Session Key**
- 用户创建 Smart Account
- 分配给 Agent 一个 Session Key：24h 有效，单笔 ≤ 0.1 ETH，日限 ≤ 1 ETH，仅白名单合约
- Agent 在限额内自动执行低风险操作
- 高风险操作弹出给用户确认
- 用户随时可撤销 Session Key

### 团队/DAO + Agent 辅助
**推荐：Safe 多签 + Agent 模块**
- Agent 生成交易草稿（提案）
- 多签成员审核并签名
- Agent 不独占控制权
- 治理流程不被绕过

### 临时/一次性操作
**推荐：EOA（但 Agent 不接触私钥）**
- 用户自己用 EOA 签名
- Agent 只负责生成交易说明和模拟结果
- 不需要 Session Key 或多签的开销

## 四、哪些操作必须人工确认

无论哪种账户方案，以下操作都必须人工确认：

| 操作 | 原因 |
|------|------|
| 大额转账（超过限额） | 资产损失风险 |
| 首次 token 授权 | 可能被恶意合约利用 |
| 无限 approve | 等于永久授权 |
| 调用未知合约 | 可能是钓鱼合约 |
| 修改账户权限/签名者 | 权限变更影响所有后续操作 |
| 跨链桥操作 | 资产转移到另一条链，不可逆 |

---

## 五、一句话总结

> **EOA 像一把钥匙开一扇门 —— 给了 Agent 钥匙就等于给了全部。智能账户像一套门禁系统 —— 可以给 Agent 一张限时、限额、限区域的门禁卡。多签像银行金库 —— Agent 可以提议开库，但需要多人到场。**
